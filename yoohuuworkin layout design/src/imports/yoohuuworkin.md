# 유휴워크인(Yuhyu Work-in) — 바이브 코딩 개발 템플릿

> **이 문서는 무엇인가요?**
> "워케이션 예약관리 서비스"를 AI(바이브 코딩)에게 시켜서 만들기 위한 **요구사항 명세서 + 지시문 템플릿**입니다.
> 서비스를 처음 만드는 분도 이 문서를 AI에게 단계별로 주면서 따라갈 수 있도록 구성했습니다.

---

## 0. 시작하기 전에 — 이 템플릿 사용법 (초보자 필독)

바이브 코딩은 "AI에게 전부 한 번에 시키는 것"이 아니라 **작은 단위로 쪼개서 시키고, 확인하고, 다음으로 넘어가는 것**이 핵심입니다.

**권장 진행 순서**
1. 이 문서 전체를 AI에게 먼저 한 번 읽힙니다. → "이 서비스의 전체 맥락이야. 지금부터 단계별로 만들 거야. 이해했으면 요약해줘."
2. **8장(개발 로드맵)** 의 단계(Phase)를 **하나씩** 시킵니다. 절대 1~10단계를 한 번에 시키지 마세요.
3. 한 단계가 끝나면 **직접 실행해보고**(로그인 되는지, 화면 뜨는지) 다음 단계로 넘어갑니다.
4. 막히면 에러 메시지를 그대로 AI에게 붙여넣고 "이 에러 고쳐줘"라고 합니다.

**초보자가 꼭 기억할 3가지**
- **MVP 먼저**: IoT 스마트홈, 실시간 채팅 같은 어려운 기능은 **맨 마지막**에. 먼저 "로그인 → 숙소 보기 → 예약" 이 한 줄기부터 완성하세요.
- **돈/보안 관련은 직접 확인**: 결제·정산·개인정보는 AI가 짜준 코드를 그대로 믿지 말고, 이 문서 7장(보안)을 기준으로 검토 요청하세요.
- **하나 만들고 하나 확인**: 한 번에 완벽한 서비스를 기대하지 마세요.

## 2. 기술 스택

> 표기 규칙: **[확정]** = 사용자가 지정함 / **[추천]** = 본 서비스에 더 적합해 추가 제안

### 백엔드 (Backend)
- **[확정] Django (모놀리식)** — 강사님 추천대로. 로그인 페이지·관리자 페이지(Django Admin)가 기본 제공되고, 보안 기능(CSRF, SQL Injection 방어, 세션 관리)이 내장되어 초보자에게 가장 안전한 선택입니다.
- **[추천] Django REST Framework(DRF)** — 모바일 앱이나 향후 확장을 대비해 일부 기능(예: IoT 제어, 채팅)을 API로 빼면 유리합니다. *단, MVP 단계에서는 Django 템플릿만으로 충분하니 나중에 도입하세요.*

### 프론트엔드 (Frontend) — 모놀리식 유지가 핵심
- **[확정/기본] Django Templates (HTML)** — 모놀리식이므로 별도 React 서버를 두지 않고 Django가 화면까지 그립니다. **초보자에게 가장 단순하고 강력한 조합입니다.**
- **[추천] HTMX + Alpine.js** — React 없이도 "버튼 누르면 화면 일부만 갱신" 같은 동적 UI를 구현. SPA의 복잡함 없이 모놀리식을 유지하면서 사용자 경험을 올려줍니다. **본 서비스에 가장 추천하는 조합.**
- **[추천] Tailwind CSS** — 디자인을 빠르게. (CSS를 잘 몰라도 클래스만 붙이면 됨)

### 데이터베이스 (Database)
- **[추천] PostgreSQL** — 개발 초기엔 Django 기본 SQLite로 시작해도 되지만, **결제·정산·예약처럼 정합성이 중요한 서비스이므로 PostgreSQL을 강력 추천**합니다. AWS RDS와 궁합이 좋습니다.

### 인증/로그인 (Auth)
- **[확정] Django 기본 인증(auth)** — 로그인/세션/비밀번호 암호화 내장.
- **[추천] django-allauth** — 카카오/네이버/구글 소셜 로그인이 필요하면 도입. **역할(기업관리자·임직원·호스트)별 분기**는 커스텀 User 모델 + 권한 그룹으로 구현(6장 참고).

### 결제/크레딧 (Payment) — 한국 환경
- **[추천] PortOne(포트원, 구 아임포트)** 또는 **토스페이먼츠(Toss Payments)** — 한국 카드·간편결제를 한 번에 연동. 초보자는 PortOne처럼 통합 연동 솔루션이 편합니다.
- 크레딧은 "충전(결제) → 차감(예약) → 환불/정산"의 **장부(트랜잭션) 모델**로 설계해야 안전합니다(6장 데이터모델 참고). *결제 연동은 반드시 테스트(샌드박스) 키로 먼저 검증하세요.*

### 실시간 채팅 (Realtime) — 임직원↔호스트 문의
- **[추천] Django Channels (WebSocket)** — Django에서 실시간 채팅 구현 시 표준. Redis를 채널 레이어로 사용.
- **[초보자 대안]** MVP 단계에서는 WebSocket 대신 **"문의 게시판 + 답변" 형태(일반 폼)** 로 먼저 만들고, 나중에 실시간 채팅으로 업그레이드하는 것을 추천합니다.

### IoT 스마트홈 (호스트 원격 관리) — 가장 난이도 높음
- **[추천] AWS IoT Core (MQTT)** — 스마트 도어락·조명·온도조절을 클라우드에서 원격 제어할 때의 표준 백본.
- **[추천] SmartThings API** 등 기성 스마트홈 플랫폼 연동 — 직접 펌웨어를 다루기 어렵다면, 시중 기기(삼성 SmartThings 등)의 공개 API를 호출하는 방식이 현실적입니다.
- **[초보자 대안 / 필수]** **1단계에서는 실제 기기 연동 없이 "Mock(가짜) 제어 화면"으로 먼저 구현**하세요. 즉, "문 열기" 버튼을 누르면 DB에 '열림' 상태만 기록. 실제 기기 연동은 서비스가 검증된 후 마지막에 붙입니다. *실물 IoT는 기기·보안 비용이 크므로 절대 처음부터 하지 마세요.*

### 비동기 작업 / 알림 (Async)
- **[추천] Celery + Redis** — 정산 일괄 처리, 예약 확정 알림(이메일/문자), 입·퇴실 시간 자동화 등 "시간이 걸리거나 예약된 작업"에 사용. *MVP 이후 도입.*

### 요약 — 추천 최종 조합
```
Django + DRF(부분) + HTMX/Alpine.js + Tailwind CSS
+ PostgreSQL + Redis
+ PortOne(결제) + Django Channels(채팅) + AWS IoT Core(IoT)
+ Celery(비동기)
```

## 3. 배포 스택 (Deployment)

> **[확정] AWS** 기반. 단, 초보자에게 AWS는 진입장벽이 높으므로 난이도별로 안내합니다.

### A. AWS 내 추천 (배운 환경 유지)
| 구성요소 | 추천 서비스 | 비고 |
|---|---|---|
| 서버(앱 실행) | **[초보자 추천] AWS Elastic Beanstalk** 또는 **AWS Lightsail** | EC2를 직접 다루는 것보다 인프라를 자동 관리해줘 훨씬 쉽습니다 |
| 서버(중급) | EC2 + Docker | 자유도 높지만 운영 부담 큼 |
| 데이터베이스 | **Amazon RDS (PostgreSQL)** | 백업·보안 자동 관리 |
| 파일/이미지 저장 | **Amazon S3** | 숙소 사진, 후기 이미지 |
| 캐시/채팅 채널 | **Amazon ElastiCache (Redis)** | 채팅·세션·Celery용 |
| 정적/이미지 배포 | **Amazon CloudFront (CDN)** | 속도·비용 최적화 |
| 도메인/보안 | **Route 53 + ACM(SSL 인증서)** | HTTPS 필수 |
| IoT | **AWS IoT Core** | 스마트홈 백본(추후) |

### B. [추천] 더 쉬운 배포 대안 (초보자에게 강력 추천)
AWS 학습 비용이 부담되면, **개발 초기에는 더 쉬운 PaaS로 배포해 빠르게 서비스를 띄우고**, 사용자가 늘면 AWS로 옮기는 전략도 좋습니다. Django 배포가 클릭 몇 번으로 끝납니다.
- **Railway** / **Render** / **Fly.io** — Django + PostgreSQL + Redis를 손쉽게 한 번에 배포. 초보자 친화적.
- **추천 전략**: ① 처음엔 Railway/Render로 빠르게 출시 → ② 트래픽·정산 규모가 커지면 AWS(Elastic Beanstalk·RDS)로 이전.

### 공통 배포 필수 사항
- **HTTPS 적용 (SSL)** — 결제·로그인 서비스이므로 필수.
- **환경변수로 비밀키 관리** — DB 비밀번호, 결제 키 등을 코드에 절대 직접 쓰지 않기(`.env`, AWS Secrets Manager).
- **`DEBUG=False`** (운영 환경) — 디버그 모드 노출 금지.

## 5. 핵심 기능 명세

### 5-1. 기업관리자 (Company Admin)
- **회사/관리자 회원가입 및 인증** (사업자 정보 등록)
- **크레딧 결제·충전**: 결제(PortOne) → 회사 크레딧 잔액 증가. 충전 내역 기록.
- **임직원 관리**: 임직원 초대(이메일)·등록·비활성화. 임직원별 사용 한도(예산) 설정.
- **사용 내역 대시보드**: 임직원별/기간별 크레딧 사용량, 예약 현황.
- **정산/세금계산서**: 월별 사용액 리포트, 영수증·세금계산서용 데이터 조회.

### 5-2. 임직원 (Employee)
- **임직원 로그인** (회사 관리자가 초대한 계정으로 가입)
- **숙소 검색·필터**: 지역, 날짜, 인원, 가격(크레딧), 편의시설(와이파이·업무공간 등) 필터.
- **숙소 상세·예약**: 사진·설명·후기 확인 → 날짜 선택 → **크레딧 차감 예약**. 잔여 한도 검증.
- **내 예약 관리**: 예약 목록, 예약 취소(환불 정책 반영).
- **후기 작성**: 이용 완료한 숙소에 별점·후기·사진.
- **채팅 문의**: 예약 전후로 호스트에게 1:1 문의(또는 MVP는 문의 게시판).

### 5-3. 호스트 (Host)
- **호스트 회원가입·인증** (정산 계좌 등록)
- **숙소 정보 등록/수정**: 사진(S3 업로드), 설명, 위치, 가격(크레딧), 편의시설, 예약 가능 일정(캘린더).
- **예약/입·퇴실 관리**: 들어온 예약 확인, 체크인/체크아웃 상태 관리.
- **IoT 스마트홈 원격 관리**: 도어락 열림/잠금, 조명·온도 제어 (1단계는 Mock).
- **채팅 응대**: 임직원 문의 답변.
- **정산 조회**: 확정된 예약 기준 정산 예정/완료 금액 확인.

### 5-4. 공통/시스템
- 회원가입·로그인·로그아웃·비밀번호 재설정
- 알림(이메일/문자): 예약 확정, 결제 완료, 입·퇴실 안내
- 관리자(운영자)용 Django Admin: 전체 데이터·신고·분쟁 관리

## 6. 데이터 모델 (주요 엔티티)

> AI에게 줄 때: "아래 모델을 Django models.py로 만들어줘. 관계(ForeignKey)와 제약조건을 정확히 반영해줘."

- **User** (커스텀): `email`, `password`, `role`(company_admin/employee/host), `name`, `phone`
- **Company**: 회사명, 사업자번호, 대표관리자(User FK)
- **CreditAccount**: `company`(FK), `balance`(현재 크레딧 잔액)
- **CreditTransaction**(크레딧 장부): `account`(FK), `type`(충전/차감/환불/정산), `amount`, `balance_after`, `related_booking`, `created_at` — **금액은 직접 수정하지 말고 항상 이 트랜잭션을 쌓아서 잔액을 계산**(정합성 핵심)
- **Employee**(또는 User의 role로 통합): `user`(FK), `company`(FK), `spending_limit`(예산 한도), `is_active`
- **Accommodation**(숙소): `host`(FK), `title`, `description`, `address`, `price_credit`, `amenities`, `is_active`
- **AccommodationImage**: `accommodation`(FK), `image`(S3)
- **AvailabilityCalendar**: `accommodation`(FK), `date`, `is_available`
- **Booking**(예약): `employee`(FK), `accommodation`(FK), `check_in`, `check_out`, `status`(요청/확정/취소/완료), `credit_used`, `created_at`
- **Review**(후기): `booking`(FK), `rating`, `content`, `images`
- **ChatRoom / Message**: 참여자(임직원·호스트), 메시지 내용, 시간
- **IoTDevice**: `accommodation`(FK), `device_type`(도어락/조명/온도), `status`, `last_controlled_at`
- **Settlement**(정산): `host`(FK), `period`, `total_amount`, `status`(예정/완료), `bookings`(M2M)

> **트랜잭션 안전성**: 예약 시 "크레딧 차감 + 예약 생성"은 반드시 하나의 DB 트랜잭션(`@transaction.atomic`)으로 묶어, 중간 실패 시 둘 다 취소되게 하세요.

## 8. 보안 요구사항 (결제·개인정보 서비스 필수)

- **Django 내장 보안 활용**: CSRF 보호, SQL Injection 방어(ORM), XSS 이스케이프 기본 유지(끄지 말 것).
- **비밀번호**: Django 기본 해싱(PBKDF2) 사용. 절대 평문 저장 금지.
- **HTTPS 강제**: `SECURE_SSL_REDIRECT=True`, 쿠키 `Secure`/`HttpOnly` 설정.
- **권한 검증**: 모든 페이지에서 "이 사용자가 이 데이터의 주인인가?"를 서버에서 재확인(URL 조작 방어).
- **비밀키 분리**: `SECRET_KEY`, DB·결제 키는 환경변수/Secrets Manager로. Git에 올리지 않기(`.gitignore`).
- **결제 검증**: 결제 금액·성공 여부는 **프론트가 아니라 서버에서 결제사 API로 재검증**(위변조 방지).
- **개인정보**: 임직원·호스트 개인정보는 최소 수집·암호화 저장. 개인정보처리방침 페이지 필요.
- **운영 설정**: `DEBUG=False`, `ALLOWED_HOSTS` 지정.
- **IoT 보안**: 기기 제어 요청은 인증된 호스트 본인 + 해당 숙소 소유 여부 검증 후에만 허용.

## 9. 개발 로드맵 (단계별 — AI에게 이 순서로 시키세요)

> ⚠️ **반드시 한 Phase씩** 시키고, 동작 확인 후 다음으로.

- **Phase 0. 프로젝트 셋업**: Django 프로젝트 생성, PostgreSQL 연결, Tailwind 설정, Git 초기화, 기본 페이지.
- **Phase 1. 인증/회원**: 커스텀 User(role 3종), 회원가입/로그인/로그아웃/비밀번호 재설정, 역할별 대시보드 분기. → *"로그인하면 역할에 맞는 화면이 뜬다"까지.*
- **Phase 2. 호스트-숙소 등록**: 숙소 CRUD, 이미지 업로드(로컬→나중에 S3), 예약 가능 캘린더.
- **Phase 3. 임직원-검색/예약(핵심)**: 숙소 검색·필터·상세, 예약 생성. *(이때 크레딧은 단순 차감으로 시작)*
- **Phase 4. 크레딧/결제**: CreditAccount·CreditTransaction 모델, 충전(결제 연동), 예약 시 차감(트랜잭션 묶기), 한도 검증.
- **Phase 5. 기업관리자 기능**: 임직원 초대·관리, 사용내역 대시보드.
- **Phase 6. 후기 + 입·퇴실 관리**: 후기 CRUD, 호스트 예약·체크인/아웃 상태.
- **Phase 7. 정산**: 호스트 정산 리포트, 기업 정산/리포트.
- **Phase 8. 채팅 문의**: (먼저 게시판형 → 이후 Django Channels 실시간)
- **Phase 9. IoT 스마트홈**: Mock 제어 화면 → (검증 후) AWS IoT Core/SmartThings 연동.
- **Phase 10. 배포·보안 점검**: 8장 보안 체크리스트 적용 후 AWS(또는 Render) 배포, HTTPS.

## 10. AI에게 줄 지시 프롬프트 예시

### 첫 시작용 프롬프트
```
나는 '유휴워크인'이라는 워케이션 예약관리 서비스를 만들 거야.
첨부한 명세서(이 문서) 전체를 먼저 읽고 전체 구조를 이해해줘.
기술 스택은 Django 모놀리식 + PostgreSQL + Tailwind CSS + HTMX 야.
나는 코딩 초보자니까, 한 번에 다 만들지 말고 'Phase 0(프로젝트 셋업)'부터
단계별로 진행할 거야. 먼저 Phase 0만 진행해줘.
각 단계마다 '내가 직접 실행해서 확인하는 방법'도 알려줘.
```

### 한 단계 시킬 때 프롬프트
```
이제 Phase 1(인증/회원)을 진행하자.
명세서 4장(권한 매트릭스)과 6장(User 모델)을 반영해서,
역할 3종(기업관리자/임직원/호스트)을 가진 커스텀 User 모델과
회원가입·로그인·로그아웃을 만들어줘.
로그인하면 역할에 맞는 대시보드로 이동하게 해줘.
파일별로 어디에 무슨 코드를 넣어야 하는지 정확히 알려줘.
```

### 에러가 났을 때 프롬프트
```
아래 에러가 났어. 원인과 해결 방법을 알려주고, 고친 코드를 줘.
(에러 메시지를 그대로 붙여넣기)
```

### 보안 점검 프롬프트 (배포 전)
```
이 문서 8장(보안 요구사항) 체크리스트 기준으로
내 코드를 점검하고, 빠진 부분을 수정해줘.
특히 결제 금액 서버 재검증과 권한 검증을 중점적으로 봐줘.
```

## 부록. 초보자가 자주 막히는 지점 미리보기
- **마이그레이션 에러**: 모델 바꾸면 `makemigrations` → `migrate` 필요. 막히면 그대로 AI에게.
- **결제 연동**: 처음엔 반드시 결제사 **테스트(샌드박스) 키**로. 실제 키는 검증 후.
- **이미지 안 보임**: 개발 단계 로컬 저장 → 배포 시 S3로 전환 필요.
- **권한 누락**: "남의 데이터가 보인다" 싶으면 8장 권한 검증부터 점검.
- **IoT는 마지막**: 절대 처음부터 실물 기기 연동 시도하지 말 것.
